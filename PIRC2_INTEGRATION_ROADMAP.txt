ROLLBACK PLAN AT EACH PHASE

Phase 0 Rollback:
  → No code changes, just environment setup
  → Easy rollback: Delete env vars, revert to HTTP

Phase 1 Rollback:
  → Keep demo auth code (already in place)
  → Config AUTH_MODE="demo" to disable PiRC2
  → Automatic fallback to old auth

Phase 2 Rollback:
  → Keep demo payment code (already in place)
  → Config PAYMENT_MODE="demo" to disable PiRC2
  → Automatic fallback to demo payments

Phase 3 Rollback:
  → Minor (security headers only)
  → Easy to revert if violation found

═══════════════════════════════════════════════════════════════════════════════

KEY SUCCESS FACTORS

1. PiRC2 Credentials First
   → Without APP_ID/SDK_KEY, cannot proceed
   → Contact Pi Core ASAP (1-2 weeks lead time)

2. HTTPS Before Phase 1
   → PiRC2 requires HTTPS
   → Dev cannot proceed without it

3. Testnet Available
   → Must have access to Pi Testnet
   → Test full flows before production

4. Code Review Every Phase
   → PiRC2 spec violations early → cheaper to fix

5. Gradual Rollout
   → Phase 1: Auth only (can work independently)
   → Phase 2: Payments (depends on Phase 1)
   → Phase 3: Polish (no dependencies)

═══════════════════════════════════════════════════════════════════════════════

DEMO MODE PRESERVATION

Keep demo auth/payments working in parallel:

  CONFIG_AUTH_MODE=demo      → Use demo auth (existing)
  CONFIG_AUTH_MODE=pi-ready  → Use Pi-ready auth (in progress)
  CONFIG_AUTH_MODE=pirc2     → Use PiRC2 auth (new)

Allows:
  - Testing new PiRC2 code without breaking existing
  - Gradual user migration from demo → PiRC2
  - Easy rollback if issues found
